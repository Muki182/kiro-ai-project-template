# Kiro AI 工程化工作流指南

本指南说明如何将 Kiro 的 spec-driven 开发模式融入 AI 系统的完整研发周期。

---

## 核心理念

传统 AI 开发的典型痛点：

```
需求（脑子里）→ 代码（文件里）→ 实现偏移（不知不觉中）
```

Kiro spec-driven 模式：

```
需求（spec 文件）→ 任务（checklist）→ 实现（代码）→ 验证（hook）→ 确认（spec 更新）
```

关键差异：**需求是持久化的、可查阅的、可被自动对照的**。

---

## 推荐工作流

### 阶段 0：项目启动（一次性）

```
1. 从本模板创建项目
2. 填写 .kiro/steering/project.md
   └── 技术栈、架构约束、命名规范
3. 确认 .kiro/hooks/ 中的触发路径与你的目录结构一致
```

### 阶段 1：需求澄清 → Spec 编写

**不要直接开始写代码**。先在 Kiro 中对话，让 Agent 帮你把模糊的需求转化为结构化 spec：

```
你：我需要实现一个视频特征提取模块，支持批量处理，性能要满足实时推理要求

Kiro：好的，我来帮你起草 spec。先确认几个问题：
      1. "实时推理"的具体指标是？（延迟/吞吐量）
      2. 输入视频的格式和分辨率范围？
      3. 与哪些上下游模块对接？
```

对话完成后，让 Agent 直接写入 `.kiro/specs/` 文件。

### 阶段 2：任务执行

按 spec 中的任务 checklist 逐条执行：

```
你：执行 TASK-02：实现 VideoFeatureExtractor 的骨架代码与接口定义

Kiro：[参照 steering/project.md 中的架构约束和命名规范生成代码]
```

**注意**：每次让 Agent 执行任务时，明确引用任务编号（TASK-XX），
Agent 会自动参照 spec 中的需求约束和验收条件。

### 阶段 3：变更触发 Hook

每次保存代码变更后，相关 hook 会自动触发：

- `doc-sync`：提示更新文档
- `spec-validate`：对照需求列表验证实现
- `test-trigger`：（接口变更时）建议补充测试

**重要**：不要忽略 hook 的输出。Hook 发现的 `⚠️` 和 `❌` 是真实的设计偏差信号。

### 阶段 4：Spec 收尾

模块完成后，更新 spec 状态：

1. 在 spec 文件头部将状态改为 `已完成`
2. 确认所有 checklist 已勾选
3. 在变更记录中添加完成日期

## 上下文管理技巧

### 大项目的 Spec 拆分策略

当单个 spec 文件超过 300 行时，考虑按**功能边界**拆分：

```
specs/
├── 01-architecture.md      # 全局架构（保持精简，< 200行）
├── 02-data-pipeline.md     # 数据处理模块
├── 03-model-core.md        # 核心模型
├── 04-training-loop.md     # 训练逻辑
└── 05-inference-server.md  # 推理服务
```

告诉 Agent 当前任务对应哪个 spec，避免加载无关上下文：

```
你：我们现在处理 03-model-core.md 中的 TASK-03.2
```

### Steering 文件的更新时机

以下情况**必须**更新 `steering/project.md`：
- 引入新的关键依赖库
- 修改目录结构规范
- 发现并记录新的架构约束
- 有重要的技术决策（填入"关键设计决策记录"表格）

## 常见问题

**Q：Hook 触发太频繁，干扰了开发节奏怎么办？**

A：调整 `trigger.patterns` 的粒度。例如，将 `spec-validate` 限制为只在
你认为"完成了一个完整功能"时手动触发，而不是每次文件保存都触发。
可以将 `type: fileChange` 改为 `type: manual`。

**Q：Spec 写得太详细，反而花了很多时间？**

A：spec 的详细程度应与模块复杂度匹配。简单工具函数只需 5-10 行 spec，
复杂的训练循环可能需要 100+ 行。经验法则：spec 的编写时间不超过实现时间的 20%。

**Q：需求在开发过程中变了，怎么处理？**

A：直接修改 spec 文件，然后告诉 Agent："spec 已更新，请重新审视 TASK-XX 的实现。"
不要在代码里悄悄改掉而不更新 spec——这正是上下文漂移的来源。

**Q：团队多人协作时，如何避免 spec 冲突？**

A：将 `.kiro/` 目录纳入 Git 版本控制。spec 文件的变更通过 PR review，
和代码变更同等对待。建议在 PR 描述中说明"对应 spec 中的哪个任务"。
